# V1 Plan: Mobile-First Draft Review and Manual Medication Management

## Summary

Build `v1` around a mobile-first medication workflow where the user can always
manage schedules directly in the UI and can also use chat to draft proposed
changes that require explicit acceptance.

The main interaction loop should be:
1. The user reviews confirmed medications in the calendar area.
2. The user can manually add, edit, or remove schedules directly from UI.
3. The user can send a text chat message to draft proposed changes.
4. When a pending proposal exists, a compact draft affordance appears above the
   composer.
5. Tapping that affordance opens a mobile-first draft editor with prefilled
   form fields.
6. The user can edit or discard individual proposed actions and must explicitly
   accept the final draft before any medication state changes happen.

This version prioritizes mobile usability, shorter review loops, and fewer
disconnected surfaces than the current pinned side-panel proposal flow.

## Key Changes

### Mobile-first chat and proposal UX

- Replace the disconnected `ProposalPanel` flow with a mobile-first pending
  draft affordance attached to the chat composer area.
- On narrow layouts, tapping the draft affordance opens a modal bottom sheet.
- On wider layouts, the same content may render in a dialog rather than a
  pinned side panel.
- Keep the composer focused on:
  - text entry
  - send action
  - pending draft entry point
  - visible but non-functional `Photo` and `Voice` affordances for future work
- The draft editor should:
  - render all pending proposal actions in a single batch review surface
  - prefill editable medication fields
  - allow removing an action from the draft before acceptance
  - treat missing-info proposals as editable forms instead of a dead-end state
  - require explicit `Accept draft` before events are emitted
- Proposal editing inside the sheet remains local UI state until acceptance.
  The stored proposal stays unchanged unless the user accepts or cancels it.

### Manual medication management

- Expand the calendar screen into the main manual management surface for
  confirmed schedules.
- Add an active schedule list below or alongside the day view on mobile in a
  scroll-friendly stacked layout.
- Support direct manual actions:
  - add a medication schedule
  - edit an existing active schedule
  - remove an existing active schedule
- Manual add and edit save directly because the user is operating in the source
  of truth UI.
- Manual removal should show a lightweight destructive confirmation dialog.
- Reuse a shared medication form for:
  - manual add
  - manual edit
  - chat proposal review and editing

### Domain and event model changes

- Add support for full schedule updates in the application layer.
- Introduce a medication command service for manual schedule mutations instead
  of overloading `ChatCoordinator`.
- Extend proposal confirmation so the app can confirm edited actions rather than
  only the originally stored proposal snapshot.
- Add `medication_schedule_updated` to the medication event family.
- Keep `medication_schedule_stopped` for removals and ended schedules.
- If an edited proposal changes the medication name of an existing schedule,
  model that as stop-old plus add-new rather than mutating identity in place.

### Persistence and projections

- Update the reducer and Drift rebuild path so projections apply:
  - accepted edited proposals
  - in-place schedule updates
  - manual adds, edits, and removals
- Preserve the core rule that unaccepted proposals never appear as confirmed
  medication state.
- Keep proposal history auditable even if the user edits the final accepted
  draft before confirmation.

### Documentation and product messaging

- Update the README to describe:
  - mobile-first draft review from chat
  - manual medication management from UI
  - explicit acceptance for AI-suggested changes
  - text-first implementation with photo and voice scaffolding only

## Public Interfaces and Types

- `ChatCoordinator.confirmPendingProposal` should accept a finalized list of
  edited actions.
- Add a dedicated medication command service with direct UI methods for manual
  add, edit, and remove flows.
- Introduce a UI-owned editable draft model separate from immutable
  `ProposalView`.
- Extend medication repository reads as needed so forms can prefill from active
  schedules and refresh immediately after writes.

## Test Plan

Detailed execution checklist:
[docs/plans/v1-testing.md](docs/plans/v1-testing.md)

- Confirm an edited proposal and verify emitted medication events use the edited
  values, not the original proposal payload.
- Confirm a batch proposal where one action is removed before acceptance.
- Apply an `update_medication_schedule` proposal and verify projections update
  correctly.
- Manually add a schedule and verify it appears in active schedules and the day
  calendar.
- Manually edit a schedule and verify updated time, dose, and notes propagate.
- Manually remove a schedule and verify it no longer appears as active while
  preserving history.
- Verify pending proposals never appear in confirmed calendar entries.
- Verify cancelling a proposal leaves medication projections unchanged.
- Add widget coverage for:
  - composer draft affordance
  - mobile draft sheet
  - shared medication form validation
  - active schedule add/edit/remove flows

## Assumptions and Defaults

- `v1` is optimized for mobile first and adapts upward for larger layouts.
- Text chat is the only functional AI input in this pass.
- Photo and voice entry points are visible scaffolding only.
- Manual UI changes are direct-write flows.
- AI-generated changes remain confirmation-gated.
- The same shared medication form drives both manual edits and draft review.
