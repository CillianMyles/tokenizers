# V0 Plan: Reliable, Event-Sourced Medication Capture with Offline-First Foundations

## Summary

Build v0 as a Flutter app with a hand-authored chat shell, Gemini-backed structured proposals, and a local event-sourced persistence model using Drift on SQLite. The app accepts text, voice, and image inputs, stores the original discussion and attachments locally, asks Gemini for conversational output plus structured medication proposals, and renders proposal review/edit/confirm UI in the app shell. The source of truth is an immutable event store; projections power chat, proposal, medication, and calendar reads.

## Implementation Progress

Status as of 2026-04-06:

- Landed:
  - routed app shell with `Chat`, `Calendar`, and `History`
  - app-owned event and model contracts
  - confirmation-gated chat -> proposal -> medication workflow
  - live Gemini provider behind the `ModelProvider` interface, with an explicit configuration error when no API key is configured
  - Drift-backed `event_log` and projection tables on native and web
  - unit tests for projection rebuild logic and command orchestration
  - GitHub Actions CI for format, analyze, and test checks on pushes and PRs to `main`
- Still open:
  - attachment, image, and voice ingestion
  - proposal editing beyond confirm/cancel
  - medication update conflict review UI
  - broader widget and integration coverage

The product loop is:
1. User sends text, image, or voice input.
2. Input and attachment metadata are written to the local event store.
3. Gemini returns conversational text plus structured proposed medication actions.
4. The app validates that output into pending proposal state.
5. User edits, confirms, or cancels.
6. Confirmation emits medication-domain events that update projections.
7. Full history remains auditable, reviewable, and available offline.

This version should optimize for reliability, high-confidence core logic, and offline-first local ownership of data. Future embedded Gemma/Gemma 4 offline inference is out of scope for v0, but the architecture should avoid coupling the app to always-online model access.

## Key Changes

### Product and UI structure

- Replace the current prompt-driven prototype shell with a real app shell built around:
  - `ChatScreen` as the primary input and discussion surface
  - `ProposalPanel` for structured review and confirmation
  - `MedicationCalendarScreen` for confirmed schedules
  - `ConversationHistoryScreen` or equivalent thread access
- Show pending proposals inline with the active thread or in a pinned review area above the composer.
- Enforce a hard confirmation gate: no medication or calendar state changes happen until explicit user confirmation.

### Event-sourcing architecture

- Use `event store + projections`.
- Make immutable domain events the only source of truth.
- Maintain projection tables for fast UI reads rather than replaying all events on every screen load.
- Include event metadata on every event:
  - `event_id`
  - `aggregate_type`
  - `aggregate_id`
  - `event_type`
  - `occurred_at`
  - `causation_id`
  - `correlation_id`
  - `actor_type` (`user`, `model`, `system`)
  - payload JSON
- Keep replay and projection rebuild as first-class workflows so every read model can be reconstructed from the event log.

Important event families for v0:
- conversation events
  - `thread_started`
  - `message_added`
  - `attachment_added`
  - `model_turn_recorded`
- proposal events
  - `proposal_created`
  - `proposal_field_edited`
  - `proposal_confirmed`
  - `proposal_cancelled`
  - `proposal_superseded`
- medication events
  - `medication_registered`
  - `medication_schedule_added`
  - `medication_schedule_updated`
  - `medication_schedule_stopped`

### Model contract and offline-first direction

- Keep one Gemini interaction loop per user turn.
- Define an app-owned, versioned response contract from Gemini with two logical outputs:
  - conversational response text
  - structured proposed medication actions
- Treat all model output as advisory only.
- Validate Gemini output before emitting proposal events.
- Proposal action types for v0:
  - `add_medication_schedule`
  - `update_medication_schedule`
  - `stop_medication_schedule`
  - `request_missing_info`
- If the model is uncertain, prefer incomplete pending proposals or missing-info requests instead of guesses.
- Store original image/audio files locally first, then pass them to Gemini as multimodal context.
- Introduce a `ModelProvider` abstraction now so Gemini is one implementation behind an interface rather than hard-coded app-wide.
- Keep the app usable offline for local history, calendar, and review of prior data even when model calls are unavailable.
- Do not implement local Gemma inference in v0, but keep the orchestration boundary compatible with a future embedded small Gemma/Gemma 4 model.

### Persistence model and projections

- Add Drift schema in two layers:
  - event store tables
  - projection/read-model tables
- Event store tables:
  - `event_log`
  - optional projection checkpoint / rebuild metadata table
- Read-model tables:
  - `conversation_threads_view`
  - `messages_view`
  - `attachments_view`
  - `proposals_view`
  - `proposal_actions_view`
  - `medications_view`
  - `medication_schedules_view`
  - `medication_schedule_times_view`
- First-class medication fields in v0:
  - medicine name
  - dose amount
  - dose unit
  - optional route
  - start date
  - optional end date
  - one or more fixed times per day
  - free-text notes/instructions
- Defer PRN medicines, tapering plans, refill workflows, household multi-profile support, and clinician sharing.

### Confirmation and domain rules

- New model suggestions always create pending proposal state, never direct mutations.
- If a new discussion implies a change to an active schedule, emit proposal events referencing the current active schedule but do not alter it until confirmation.
- Confirmation emits medication-domain events which then update projections.
- Cancelling a proposal emits cancellation events and preserves the audit trail.
- Stopping a medication ends the active schedule through events; it never deletes history.
- If a proposal conflicts with the current projected schedule, surface `current vs proposed` in the confirmation UI before approval.

### Reliability, correctness, and quality gates

- Put critical business logic in pure Dart domain/application layers, not inside widgets.
- Isolate the following as independently testable units:
  - event creation and validation
  - projection reducers
  - proposal interpretation/validation
  - medication conflict detection
  - schedule derivation for calendar display
  - model response parsing and unavailable/error behavior
- Prefer deterministic reducers and command handlers so replay and regression testing are straightforward.
- Treat malformed or partial model output as a normal failure mode with explicit handling:
  - preserve the raw model turn in the event log
  - emit no medication-changing events
  - surface a recoverable UI state
- Make projection rebuild part of reliability strategy, not an afterthought.

### Linting, tests, and CI

- Expand static analysis beyond the default Flutter lints by adopting stricter project lints appropriate for a reliability-focused app.
- Add CI for:
  - pushes to `main`
  - pull requests targeting `main`
- CI pipeline should run, in order:
  - dependency install / Flutter setup
  - `dart format --output=none --set-exit-if-changed .`
  - `flutter analyze`
  - `flutter test`
- Add separate test suites where helpful:
  - unit tests for domain/event/projection logic
  - repository tests for Drift-backed persistence behavior
  - focused widget tests for proposal confirmation and calendar rendering
- Prioritize high coverage for important logic rather than vanity global coverage:
  - event reducers
  - command/application services
  - model contract parsing
  - confirmation gate behavior
  - schedule/calendar derivation
- Keep coverage reporting in CI so regressions are visible, even if v0 does not enforce a hard numeric threshold yet.
- Add a small set of golden-path integration tests around:
  - text -> proposal -> confirm
  - image -> proposal -> edit -> confirm
  - update existing medication -> conflict review -> confirm
- Keep GitHub Actions simple and deterministic; no mutation or auto-format-on-CI behavior.

### Calendar and review UX

- Calendar scope for v0 is a day-based view of confirmed medication times with simple date navigation.
- Pending proposals never appear as confirmed calendar entries.
- Every visible medication/schedule should be traceable back to:
  - confirming proposal
  - originating discussion thread
  - original attachments/messages where applicable
- Conversation history should expose both the readable thread and the reviewable lifecycle:
  - what the user said
  - what the model proposed
  - what the user edited
  - what was confirmed or cancelled

## Test Plan

- Create medication from text:
  - emit message/model/proposal/confirm events
  - verify medication and calendar projections reflect the confirmed schedule
- Update existing medication from text:
  - verify no active schedule changes before confirmation
  - verify confirmation emits update events and projection changes
- Stop medication:
  - verify stop event closes schedule and preserves prior history
- Image/script ingestion:
  - verify local attachment reference stored as an event
  - verify proposal generated from that turn and remains editable before confirm
- Voice-note ingestion:
  - verify audio file reference is stored
  - verify resulting proposal can be confirmed without losing source linkage
- Uncertain extraction:
  - verify missing-info or incomplete proposal state rather than accidental schedule creation
- Cancel flow:
  - verify cancellation event exists and no medication/calendar projection changes occur
- Projection rebuild:
  - replay event log into empty projections
  - verify chat history, proposals, schedules, and calendar state are restored exactly
- Offline behavior:
  - verify history and calendar remain usable without network
  - verify model-unavailable state is explicit and non-destructive
- Conflict case:
  - existing active schedule plus contradictory proposal
  - verify review UI shows the conflict before confirmation
- App restart:
  - verify projections reload correctly from local DB without contacting Gemini
- Malformed model payload:
  - preserve raw model turn in event log
  - emit no medication-changing events
  - show recoverable failure state in UI
- CI validation:
  - formatting, analysis, and tests run on PRs to `main` and pushes to `main`

## Assumptions and Defaults

- Flutter remains the client stack and Gemini remains the initial model provider.
- Drift/SQLite is the only storage layer in v0; there is no backend.
- Event store plus projections is the required persistence model.
- Cross-platform parity remains the intent, with graceful degradation where capture UX differs.
- Text input is mandatory everywhere; image uses file selection; voice uses locally recorded files under a common attachment abstraction.
- Notifications remain out of scope for v0.
- The app should be architected as offline-first for storage and review, even though model inference remains online in v0.
- Future embedded small Gemma/Gemma 4 inference is a design constraint, not a v0 deliverable.
- The app is a patient-side coordination and tracking tool, not a diagnosis or autonomous treatment tool.
